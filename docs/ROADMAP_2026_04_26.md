### Cross-session protocol (avoid stepping on each other)

1. **Each session works on a feature branch**: `stage1-w4a8`, `stage2-e4b`,
   `stage3-prefill-bn`, `stage5-qwen3vl-linear`, etc.
2. **First action**: `git checkout main && git pull && git checkout -b <stage>`.
3. **In-flight tasks**: append a one-line entry to `docs/INFLIGHT.md`
   with `branch | session-id | started | task` so other sessions don't
   duplicate. Remove the line when the branch merges.
4. **Merge protocol**: rebase onto current `main`, push to remote feature
   branch, open PR (or fast-forward merge if user authorises and CI is
   green). After merge, delete feature branch + remove INFLIGHT entry.
5. **Conflict policy**: `Sources/CoreMLLLM/Gemma4StatefulEngine.swift`
   and `Examples/CoreMLLLMChat/CoreMLLLMChat/LLMRunner.swift` are shared
   hot files. If two stages both touch them, the one to merge first
   wins; the other rebases.

## 2. Stage detail

### Stage 1 — W4A8 via `linear_quantize_activations` (cml9 PR #2577 final form)

**Goal:** ship INT8 activation-quant on top of the validated `nn.Linear`
projection path. Halves intra-op memory bandwidth on Apple ANE.

**Why now:** Plan 3 confirmed Linear stays 100 % on ANE. cml9
`linear_quantize_activations` fires on `linear` ops (PR #2577 main
contribution). We have all prerequisites; nobody else has tried this on
Gemma 4 stateful.

**Expected gain:** decode +30-50 % if memory-bandwidth-bound at decode
(Gemma 4 E2B is). 40 tok/s → 50-60 tok/s on iPhone 17 Pro.

**Risks:** quality regression from 8-bit activations (quantile-aware
calibration may be needed); ANE may decompose to non-fused ops.

**Steps:**
1. PoC on chunk_1 only with `--linear-projections` + new
   `--activation-quant` flag. Build. ANE-audit. Mac latency probe.
2. Numerical comparison against W4 (no activation quant) chunk_1 —
   max-abs diff and PPL on a small sample.
3. If PoC OK, build full 4-chunk W4A8 bundle. Mac sanity. iPhone push +
   measure. Compare to current Linear+W4 baseline.
4. Verdict: ≥ +20 % decode tok/s and PPL parity → ship; otherwise HOLD.

**Files to touch:**
- `conversion/build_gemma4_e2b_stateful_chunks.py` — `--activation-quant`
  flag, dispatches `cto.coreml.experimental.linear_quantize_activations`
  after palettize.
- `conversion/probe_w4a8_quality.py` (new) — PPL on a fixed sample to
  catch regression.

**Branch:** `stage1-w4a8`. ETA 1-2 days.

### Stage 2 — Gemma 4 E4B stateful port

**Goal:** apply the Phase 1 + 2a wins to E4B (the larger 4 B sibling).

**Why now:** E4B is structurally identical to E2B; the Linear-projections
flag and stateful chunk classes already support it (chunk boundaries
parametric). Pure data-only port.

**Expected gain:** E4B chat connects with TTFT -95 % on multi-turn.
Decode tok/s unchanged from current 14 tok/s on iPhone 17 Pro (limited
by DRAM bandwidth, not compute) — but combined with W4A8 (Stage 1)
might see real decode lift if memory bandwidth was the limit.

**Risks:** chunk boundary differences (E2B = L0-7/L8-14/L15-24/L25-34,
E4B = L0-11/L12-22/L23-32/L33-41 — actual numbers in
`Gemma4Config.compute_chunk_boundaries`). Bundle is bigger.

**Steps:**
1. Build E4B stateful bundle: `python3.12
   conversion/build_gemma4_e2b_stateful_chunks.py --model gemma4-e4b
   --output /tmp/gemma4-e4b-stateful --linear-projections`.
2. Mac sanity (same script, swap path).
3. Add `gemma4e4bStateful` ModelInfo entry to `ModelDownloader.swift`
   (mirror `gemma4e2bStateful`); `localModelURL` already detects the
   subdir layout, no change there.
4. Assemble bundle (re-use `assemble_gemma4_stateful_bundle.sh` with E4B
   sidecars from `staging-2k-fast-prefill/gemma4-e4b/`).
5. iPhone push + test. Same hello / chat-continuation pattern.

**Files to touch:**
- `Sources/CoreMLLLM/ModelDownloader.swift` — one ModelInfo entry.
- `scripts/assemble_gemma4_stateful_e4b.sh` (new, copy of bundle script
  pointing at E4B sources).

**Branch:** `stage2-e4b`. ETA 1 day.

### Stage 3 — Phase 2b: multifunction `prefill_bN`

**Goal:** first-turn TTFT win via batched (T = 8 / 16) prefill.

**Why now:** Phase 2a addressed the multi-turn case; long single-turn
prompts still re-prefill linearly at 38 tok/s. Multifunction is the
remaining lever to drop that to ~150-200 tok/s.

**Expected gain:** 50-tok prompt prefill 1.2 s → 0.16 s (-87 %).

**Risks:** multifunction + state coexistence is non-trivial. Reference
impl is Qwen3-VL v1.5.0 (`build_qwen3_vl_2b_stateful_multifunction.py`
and `Qwen3VL2BStatefulGenerator.swift:518-562`); Gemma 4's ISWA + dual
state needs careful slice_update for T = 8 ring writes.

**Steps:**
1. Add `--prefill-batches "8,16"` flag to converter, emit
   `prefill_b8` / `prefill_b16` functions alongside `infer`.
2. Engine `load()` probes for multifunction; `generate()` dispatches
   `prefill_bN` when input length ≥ N, falls back to T = 1 for the tail.
3. Bundle re-deploy + iPhone validation.

**Files to touch:**
- `conversion/build_gemma4_e2b_stateful_chunks.py` — multifunction emit.
- `conversion/models/gemma4_swa_stateful_chunks.py` — T = N forward.
- `Sources/CoreMLLLM/Gemma4StatefulEngine.swift` — function dispatch.

**Branch:** `stage3-prefill-bn`. ETA 2-3 days.

### Stage 5 — Plan 3 migration to other converters

**Goal:** drop `ane_ops.Conv2dLinear` once Qwen3-VL / Qwen3.5 /
4-chunk Gemma 4 converter paths are also Linear-validated.

**Why later:** Stages 1-3 deliver the headline wins; Stage 5 is code
cleanup + future-proof. Worth doing once we have W4A8 confidence.

**Steps:**
1. For each converter (Qwen3-VL 2B, Qwen3.5 0.8B / 2B, 4-chunk Gemma 4):
   add `--linear-projections` flag using the same `_project` helper
   pattern as `gemma4_swa_stateful_chunks.py`.
2. Mac probe per converter (`probe_e2e_linear_latency.py` adapted).
3. iPhone validation per converter.
4. Once all paths Linear-validated and shipped, delete `ane_ops.Conv2dLinear`.

**Branch:** `stage5-qwen3vl-linear` (Qwen3-VL first), then
`stage5-qwen35-linear`, `stage5-gemma4-4chunk-linear`. ETA 1-2 days
each.

## 3. HF upload + production swap + release timing

### Phase A — current state (today)

- HF: only the legacy 4-chunk Gemma 4 E2B / E4B bundles are uploaded.
- App: stateful entries are sideload-only behind `LLM_SHOW_EXPERIMENTAL=1`.
- No production user sees the new path.

### Phase B — post-Stage 1 + Stage 2 (target: 1 week from today)

When **W4A8 is validated and E4B is ported**:

1. **HF upload** — push the stateful bundles as HF releases:
   ```
   huggingface-cli upload mlboydaisuke/gemma-4-E2B-coreml \
       build/gemma4_stateful_ab/linear/gemma4_e2b_stateful_chunks/ \
       --repo-type=model
   ```
   (or whatever the existing HF release pattern is — re-use the path
   that `ModelDownloader.swift` lines 819-898 expect.)
2. **Wire `downloadURL`** in `gemma4e2bStateful` / `Linear` ModelInfo to
   the new HF location; remove the empty-string sideload-only marker.
3. **Keep `LLM_SHOW_EXPERIMENTAL=1`** gating until iPhone-wide A/B with
   real users (e.g. you on chat for a few days).

### Phase C — production swap (target: 2 weeks from today)

1. **Move the stateful entries OUT of the experimental gate** in
   `ModelDownloader.swift:defaults`. Make them visible by default.
2. **Pick a primary**: stateful Linear becomes the default Gemma 4 E2B /
   E4B in the picker. The 4-chunk recurrent path stays as
   `gemma4-e2b-legacy` (or similar) for one release cycle, then drops.
3. **README** update — `Gemma 4 E2B` rows in the perf table swap from
   "32 tok/s decode" to "40 tok/s decode + 0.34 s 2nd-turn TTFT".

### Phase D — release tag

After production swap soaks for ~1 week:
- `git tag v1.7.0 -m "Gemma 4 stateful + Phase 2a cross-turn KV reuse"`
- README headline + the changelog entry call out the TTFT win as the
  marquee feature.
