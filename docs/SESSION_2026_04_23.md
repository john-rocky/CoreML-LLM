**Scope:** TTFT + load-time levers inspired by Google AI Edge Gallery / LiteRT-LM.
Major research pivot mid-session when a Mac-vs-iPhone bench revealed an
undocumented iPhone ANE property that invalidates one of our top
optimization candidates.

**Start here next session:** this file + `IPHONE_ANE_SPARSITY_FINDING.md`

---

## The session-level finding (top)

iPhone 17 Pro A19 Pro ANE runs Gemma 4 prefill compute **proportional to
`real_len` (valid tokens), not to `N` (static batch dim)**. Mac Studio
ANE, by contrast, runs full-N compute regardless of mask.

- Mac bench (prefill_chunk1, cpuAndNeuralEngine): b64=4.6ms, b128=9.1ms,
  b256=19.8ms, b512=46.0ms — ratio 0.10x matches compute ratio 0.125
- iPhone b512 graph: 75ms for real_len=10, 560ms for real_len=348 —
  scales with real_len

**Consequence:** optimizations that reduce N (multi-length prefill
variants, dynamic reshape) give **zero iPhone gain**. Optimizations that
reduce real_len (prefix caching, compression) give direct seconds back.

Full detail: `docs/IPHONE_ANE_SPARSITY_FINDING.md`.

## Explored and reverted (results in docs)

### Multifunction prefill variants (reverted)

- Built `conversion/build_prefill_multifunction.py` that packs
  `prefill_b64/128/256/512` into one mlpackage with 75% weight dedup
  via `ct.utils.save_multifunction`.
- Added Swift router (`PrefillSet`, `pickPrefillSet`) in
  `ChunkedEngine.swift`, opt-in via `LLM_PREFILL_MULTIFUNCTION=1`.
- **Mac bench confirmed the idea works** (linear scaling with N).
- **iPhone bench killed it:**
  - Short prompt: 71ms (variant) vs 75ms (default) → 5% improvement only
  - Long prompt (348 tok): b512 fallback regressed 560ms → 1250ms when
    multifunction was loaded, ANE state contention between 4×4 MLModel
    instances
  - Just loading the multifunction mlpackage (without variants active)
    slowed finalPrewarm prefill ~3×
- Reverted Swift router (`557e71b`). Conversion + bench scripts kept as
  research artifacts — still useful on Mac or future ANE revisions.

### PrefixCache default-on (reverted)

- Flipped to default-on (`fe0f4a9`), reverted (`d471a7f`).
- Why flipped: iPhone compute ∝ real_len, so cached tokens = real seconds.
- Why reverted: cache's restore path runs per-token decode (33 ms/tok)
  over the delta. iPhone prefill is 1.4 ms/tok marginal. **Cache wins
  only when matchLen/realLen ≥ 0.95** — narrow regime.
- Break-even math table in `IPHONE_ANE_SPARSITY_FINDING.md`.
- Stays opt-in (`LLM_PREFIX_CACHE=1`) — useful for very long system
  prompts with tiny per-turn deltas; not a universal default.

## Conventions / lessons codified

1. **Mac-first validation for prefill-path work.** Mac ANE has no
   sparsity-aware optimization, so it exposes N-scaling behavior that
   iPhone hides. Bench on Mac **before** device-side integration.
   Memo in `memory/project_iphone_ane_sparsity.md`.
2. **`feat/litert-perf-adoptions` branch stays shelved.** Big bundle of
   S1/S2/T1-T5 LiteRT adoptions was empirically zero-lift on device by
   its own admission. Cherry-pick narrowly with verification, don't
   merge. Memo in `memory/project_litert_perf_branch_shelved.md`.
3. **Padding is ~free on iPhone ANE.** When sizing prefill graphs,
   prioritize coverage over wasted compute.
4. **Two regression patterns learned:**
   - Loading multiple MLModel instances from the same multifunction
     mlpackage can induce ANE state contention
   - Per-token decode is ~20× slower than batched prefill marginal
     cost on iPhone; cache designs that fall through to per-token
     decode need near-perfect match to win

## Next session — recommended order

### #1 — Decode chunk pipelining (decode-side lever; ~1 week)

**Goal:** 30 tok/s → 35-37 tok/s steady state by overlapping CPU prep of
chunk N+1 with ANE compute of chunk N. Currently sequential per
`BASELINE_SPEED_AUDIT.md:59-64`.

**Where to start:** `ChunkedEngine.predictStep` in `ChunkedEngine.swift`
— split into submit/complete phases, ping-pong scratch buffers so
next-step input prep can run while current step's ANE call is waiting.
See T3 in the shelved branch for prior sketch (was deferred there).

**Risk:** medium — ping-pong scratch buffer pool (`ChunkedEngine.swift:66-74`)
is currently single-buffered.

**Mac validation:** probably pointless — decode is latency-bound per
chunk and iPhone/Mac ANE dispatch timings differ. Better to instrument
iPhone with `os_signpost` for cpu_active vs ANE_wait ratios per chunk.

### #2 — Global-layer within-layer K=V alias (long-ctx lever; ~1 week)

**Goal:** Gemma 4's global attention layers have K=V by design
(`GEMMA4_ARCHITECTURE_VERIFIED.md`). Our export emits separate `kv14_k`
and `kv14_v` tensors. Aliasing them halves global-layer KV memory /
bandwidth — meaningful at long context (2K+, more at 8K).

**Where to start:** `conversion/models/gemma4_swa_chunks.py` — SWAChunk2
outputs, then update `ChunkedEngine.swift` KV cache allocator to share
buffers for L14 outputs.

**Mac validation possible:** yes, check conversion output correctness
via parity test before device push.

### #3 — Prefill N=1024 extension (padding-is-free lever; 2-3 days)

**Goal:** ship one prefill variant at N=1024 (instead of N=512). On
iPhone the padding cost is ~zero, so short prompts stay fast, and
medium-length prompts (513-1024) stop falling through to per-token
decode. The per-token fallback for 500-token prompts is currently
~16 seconds; batched prefill would hit ~1.5 seconds.

**Where to start:** `conversion/models/gemma4_prefill_chunks.py`
`PREFILL_N = 1024` + rebuild single-variant mlpackage. No Swift change
needed if we use the same single-function path.

**Mac validation needed:** check attention O(N²) doesn't balloon at
N=1024. The global layers are 7 of 35 and each is O(N²); at N=512 that's
already significant, at N=1024 it's 4× the global-layer compute. Need
to measure if total prefill time stays acceptable.

### #4 — PrefixCache delta-batched-prefill (future, larger surgery)

**Goal:** change PrefixCache's restore path to batched-prefill the
delta tokens instead of per-token decode. Would shift break-even from
`delta < realLen/22` to `delta < 20× bigger threshold` — cache would
win for virtually any match ≥ 20% of realLen.

**Complexity:** requires a prefill graph that can start from a
non-zero position with the restored KV state as input. Not trivial —
involves adding a position offset input, rewiring causal-mask and RoPE
tables.

**Punt unless:** there's demand for agent/tool-using workloads where
the cache pattern shows up naturally.

### Anti-list — do NOT attempt

- **Multi-length prefill variants** (any form). See this doc §reverted.
- **Short-prompt-specific TTFT optimizations.** iPhone ANE already
  handles these via real_len-aware compute.
- **Default-on flipping of PrefixCache** without first landing the
  delta-batched-prefill fix.
- **Merging the shelved `feat/litert-perf-adoptions` branch.** S2, T1
  measured as zero-lift on device; others depend on dead paths.
