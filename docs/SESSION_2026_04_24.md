## The latent bug — `writeSlidingFromPrefill` @ `ChunkedEngine.swift:2047-2065`

### Symptom

iPhone EXC_BAD_ACCESS on the 3rd inference of a fresh chat session
(N=1024 prefill shipped). Stack:

```
0 ChunkedEngine.writeSlidingFromPrefill(src:name:cache:slot:realLen:hd:)
1 ChunkedEngine.runPrefill(...)
2 closure … in CoreMLLLM.streamFromPrompt(...)
```

Line 2063:
```swift
for j in 0..<hd { dst[dstOff + j] = s[srcOff + j] }
```

### Root cause

```swift
let W = config.slidingWindow         // 512 for Gemma 4
let startCachePos = W - realLen      // NEGATIVE when realLen > W
for p in 0..<realLen {
    let dstOff = slot * slotStride + (startCachePos + p) * maxHd
    for j in 0..<hd { dst[dstOff + j] = s[srcOff + j] }
}
```

When `realLen > W`, `startCachePos` is negative and the initial
iterations write to `dstOff < 0`.

N=512 guaranteed `realLen ≤ N = 512 = W`, so `startCachePos ≥ 0` was
incidentally preserved. N=1024 exposes the bug as soon as a prompt
(including multi-turn chat context) exceeds 512 tokens. Session log
shows the 3rd inference was the crash — likely first turn to push
cumulative context past 512.

### Secondary concern — `writeFullFromPrefill` @ 2067-2083

```swift
let slotStride = 1 * ctx * maxHd  // ctx=2048
for p in 0..<realLen {
    let dstOff = slot * slotStride + p * maxHd
    ...
}
```

Here `realLen ≤ N = 1024` and ctx=2048, so `dstOff` stays in range.
**This one is safe at N=1024.** Only becomes unsafe if prompts exceed
ctx=2048, which is a separate issue.

### Why the N=1024 Mac bench didn't catch this

Mac bench (`benchmark_prefill_multifunction.py`) calls `predict` on
dummy inputs — it measures mlpackage latency. It never exercises the
Swift-side `writeSlidingFromPrefill` post-processing. So Mac-first
validation is blind to Swift-KV-write bugs by design.

**Lesson for the iPhone testing lane:** mlpackage validation is
necessary but not sufficient — Swift post-processing paths must be
exercised on device. A quick pre-ship check: load N=1024 mlpackage on
device, run a **single** >512 token prompt, confirm no crash.

### Fix plan (next session)

The write target is the SWA decode cache. Before writing the fix,
**read the decode side** to understand which cache index represents
which absolute position:

- `ChunkedEngine.swift` decode path — how does decode at position p
  write the sliding KV cache? By `p % W`? Or via `updateIndicator`
  scatter?
- Is the prefill "right-align" layout (`startCachePos = W - realLen`)
  consistent with that decode convention when realLen ≤ W?

Only after the convention is clear: update `writeSlidingFromPrefill`
to handle `realLen > W` by writing only the last W positions of the
source (skip `sourceStart = realLen - W` positions), using whatever
cache-index mapping decode expects.

Likely fix shape:

```swift
let writeCount = min(realLen, W)
let sourceStart = realLen - writeCount  // 0 when realLen ≤ W
let startCachePos = W - writeCount      // 0 when realLen ≥ W
for p in 0..<writeCount {
    let srcOff = (sourceStart + p) * hd
    let dstOff = slot * slotStride + (startCachePos + p) * maxHd
    ...
}
```

But `startCachePos` vs `p % W` convention must be confirmed before
shipping.

Estimated effort: 30 min read + 30 min code + 30 min iPhone verify.
Mac rebuild not needed (N=1024 mlmodelcs are already on disk).
