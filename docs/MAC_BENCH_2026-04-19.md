# Mac bench after LiteRT-perf adoptions (2026-04-19)

Captured immediately after landing the runtime changes described in
`LITERT_PERF_ADOPTIONS.md` (T1 LRU, T2 gate, T3 QoS, T4 cancel, T5
hysteresis, S1 router, S2 prefetch). The model is *not* yet on device —
all numbers below come from a Mac with on-host ANE.

## Environment

- macOS Darwin 25.0.0, Xcode 26.1.1, Swift 6.2.1
- Model: `~/Downloads/gemma4-e2b-eagle3-sideload` (Gemma 4 E2B, ctx=2048)
  - Has decode chunks (`chunk{1..4}.mlmodelc`) but **no prefill chunks**
    and verify is in separate legacy `verify_chunk*.mlmodelc` files
    (the runtime expects `verify_qK` as a multi-function inside
    `chunk*.mlmodelc`, so spec is disabled here)
- Build: `swift build -c release`
- Bench: `swift run -c release coreml-llm-smoke <dir> "<prompt>" <maxTokens>`

### What the numbers say

- **Steady-state decode = ~33 tok/s** on Mac (this hardware/model).
- **Mask build cost dropped from 0.4 ms to 0.0 ms** between prewarm and
  steady state. That is S2 working: the mask + RoPE for position N+1
  is built on `prepQueue` while chunks 1-4 of position N are running on
  ANE, so the foreground reads it as a hit on the next iteration.
- **Embedding cost dropped from 0.5 ms to ~0.0 ms.** That is T1 working:
  the LRU hits for the small set of repeated subword tokens in the
  output. The savings are modest (vDSP was already fast) but they come
  off the P-core, which matters for thermals on long sessions.
- The first post-prefill step (`30.0 ms`) is already at steady state,
  not the higher prewarm number. The `schedulePrefetch(forPosition:
  realLen)` call we added at the end of `runPrefill` ensures the first
  decode token starts with hot prefetched buffers.

### A/B comparison (added 2026-04-19, post-bench)

To get an honest delta, both T1 and S2 were toggled off via env vars
(`DISABLE_EMBED_LRU=1 DISABLE_PREFETCH=1`) and the same 128-token run
was repeated:

| condition | tok/s | total ms/step | emb | mask |
|---|---|---|---|---|
| BASELINE (T1 + S2 off) | 33.17 | 30.2 | 0.3 ms | 0.0-0.1 ms |
| OPTIMIZED (T1 + S2 on) | 33.21 | 30.0 | 0.0 ms | 0.0 ms |

**Mac decode tok/s is essentially unchanged (+0.1 %, within noise).**
The reasons:

1. CPU prep was already tiny (~0.4 ms) on a 30 ms ANE step, so even a
   perfect hide caps the speed-up at ~1.3 %.
2. Mac has plentiful idle CPU during ANE waits, so the "uncached"
   condition still completes its prep in real time without blocking
   the next ANE submit.
3. The ANE itself is the rate limiter (~29.8 ms / step regardless).

These optimisations were not designed for Mac peak tok/s. Their target
is iPhone:
- T3 (.utility QoS) + T1 (LRU) keep work off P-core, which matters
  more on iPhone where waking a P-core for 1 ms is the worst joule
  ratio.
- S2 hides CPU prep behind ANE work — visible as time only when
  CPU prep is non-trivial relative to ANE step time.
- Sustained iPhone runs hit thermal throttle; reducing CPU op count
  delays the throttle point.

What the Mac bench *does* prove:
- The prefetch and LRU paths are functionally correct (mask = 0.0 ms,
  emb = 0.0 ms when enabled).
- Nothing regressed: base path still hits 33 tok/s.
- The ANE residency CI gate (T2) is working and revealed a const-op
  filter bug that was masking the true ~99 % residency as ~93 %.

### Per-step decomposition (steady state, 30 ms total)

```
emb (CPU, T1 LRU):       0.3 ms   (was ~0.5 ms cold)
mask + RoPE (S2 prefetch): 0.0 ms   (was ~0.4 ms cold; built behind ANE work)
chunk1  (ANE):           5.3 ms
chunk2  (ANE):           6.6 ms
chunk3  (ANE):           7.4 ms
chunk4  (ANE + argmax):  10.3 ms
─────────────────────────────────
total:                  ~29.9 ms  → 33.5 tok/s
```

Approximately **0.6 ms / step (≈2 %) of CPU work was hidden** by S2 +
T1 vs the cold path. On Mac that is a small steady-state win because
the CPU was already keeping up. On iPhone the same change moves work
off P-core entirely (T3) and matters more for sustained thermal
behaviour.

## What we did NOT bench yet

- **S1 prefill variants:** this model directory has no `prefill_chunk*`
  at all (default OR variants), so the runtime falls back to per-token
  decode for the prompt. Re-export prefill chunks at multiple sizes via
  `conversion/build_prefill_variants.py` once an HF checkpoint is on
  the host.
- **T5 hysteresis gate:** verify is not loaded for this model (the file
  layout has `verify_chunk*.mlmodelc` as separate models, but the
  current runtime expects `verify_qK` as a multi-function inside
  `chunk*.mlmodelc`). DrafterUnion is therefore inert in this run.
- **T4 cancellation:** smoke does not cancel; needs an interactive run.
  The hooks are wired in `CoreMLLLM.swift` (`onTermination` →
  `genTask.cancel()`) but a UI test is the right way to validate.

## Xcode setup

The Swift Package opens directly in Xcode:


```
Schemes:
    accept-rate-bench
    ane-residency-gate           ← new
    coreml-llm-smoke
    CoreMLLLM
    CoreMLLLM-Package
    union-bitexact
    video-test
```

Run any scheme from Xcode (Product → Run, or ⌘R after picking the
scheme). For CLI tools that take args (`ane-residency-gate`,
`coreml-llm-smoke`, etc.) edit the scheme's Arguments tab.

`swift build -c release` from the package root mirrors what Xcode
produces; CLI workflows can stay terminal-only without losing parity.

## Baseline file

`/tmp/ane_residency_baseline.json` is the per-chunk ANE residency
snapshot from this run. Pin it in CI (or copy into the repo) and pass
it to subsequent runs via `--baseline` to detect regression:

```
swift run -c release ane-residency-gate \
  --model-dir <new dir> \
  --threshold 0.985 \
  --baseline ane_residency_baseline.json \
  --write-baseline ane_residency_current.json
```
